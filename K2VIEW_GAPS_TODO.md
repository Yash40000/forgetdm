- [x] Micro-DB industry-standard hardening pass (migration `V50__business_entity_capsule_hardening.sql`):
  - **Enforced access grants** — grants are no longer evidence-only: `detail`/version reads need READ, provision needs PROVISION, refresh/retire/grant/restore need MANAGE (scope hierarchy READ < PROVISION < MANAGE < OWNER), matched by username or role, honoring `expires_at` and revocation; `admin.all` bypasses; the materializing user gets an implicit OWNER grant; unauthorized access is a 403 (`ApiException.forbidden`).
  - **Encryption at rest** — MASKED fragment payloads are AES-256-GCM ciphertext under a per-capsule key derived from `forgetdm.capsule-secret` + a random per-instance `key_salt` (`CapsuleCrypto`), per-fragment random IV; decrypted only for access-checked readers via never-persisted copies; content hashes stay over the clear payload so versions remain comparable.
  - **Restorable version history (time travel)** — fragments carry `version_no` and superseded payloads are retained; `GET /capsules/{id}/versions/{v}/fragments` reads any version; `POST /capsules/{id}/versions/{v}/restore` re-issues an old version as a NEW current version (history immutable).
  - **Truncation evidence** — the 500-row capture cap now fetches cap+1 and records a `truncated` flag + message instead of clipping silently.
  - **Concurrency safety** — materialize serializes on a pessimistic row lock (`findWithLockByEntityIdAndCanonicalKey`); a create/create race surfaces as a clean 409 instead of a constraint explosion or double-incremented versions.
  - **Sync-on-demand (K2View "sync on access")** — capsules carry `sync_mode` (MANUAL/ON_DEMAND) + `stale_after_minutes`; `GET /capsules/{id}` transparently re-materializes a stale ON_DEMAND capsule (also when any watermark reports STALE), with `SYNC_ON_DEMAND`/`SYNC_FAILED` lineage.
